# 产品需求文档：AI极简生活清单

**版本**：V1.0

**目标平台**：Web App（H5）

**更新日期**：2026-01-15

**产品定位**：一款不需要学习分类的清单工具。用户像发消息一样输入，AI 自动分类整理。

---

## 1. 产品概述

- **一句话定位**：你只管说，AI 帮你整理。
- **目标用户**：上班族（日程 + 采购）、宝妈（购物 + 育儿），同时兼顾全人群的轻量记录场景。
- **核心价值**：省去手动分类与手动选择时间的操作，从“管理清单”回归到“记录生活”。支持移动端浏览器访问、数据云端存储（单设备内持久化），隐私优先，无需登录。
- **产品差异化**：零操作门槛 + AI 自动分类 + 极简界面 + 面向移动端的低摩擦交互。

---

## 2. 功能列表（优先级）

| 优先级 | 功能模块 | 描述 |
| --- | --- | --- |
| P0 | AI 语义识别输入 | 输入一句话，自动识别分类（购物 / 日程 / 其他）和时间 |
| P0 | 清单展示与分类切换 | 按分类（全部 / 购物 / 日程 / 其他 / 已完成）展示事项列表 |
| P0 | 事项操作 | 添加、标记完成 / 未完成、删除（含撤销）、修改分类、修改时间 |
| P0 | 云存储 | 所有事项通过后端 REST API 存储到 MongoDB，同一浏览器（设备）内数据持久化 |
| P1 | 用户偏好设置 | 选择“上班族”或“宝妈”模式，影响默认排序 |
| P1 | 动态标签 | 根据关键词自动生成“工作”“育儿”等子标签，支持隐藏 |
| P2 | 智能提醒 | 识别时间后，到点发送浏览器通知或第三方提醒 |
| P2 | 模糊检索 | 搜索框，按关键词查找事项 |

**V1.0 实现范围**：P0 全部 + P1 全部；P2 延后至 V2.0。

## 3. 用户故事

- **作为上班族**，我希望在早上快速记录“今天下午 3 点开会，买咖啡”，系统自动拆解和归类，这样我不用手动选择分类和时间。
- **作为宝妈**，我随时想起“买奶粉、周三接孩子”，系统能自动归为购物和日程，并且购物类更靠前，方便我临时查看。
- **作为普通用户**，我删错了事项，希望能立刻撤销，避免重新输入。
- **作为移动端用户**，我希望整个交互像手机里的消息输入一样自然，输入、长按、修改都不需要学习。
- **作为注重隐私的用户**，我希望我的数据被隔离保存，不依赖第三方社交账号登录，不同设备间数据互不影响。

## 4. 页面与交互详情

### 4.1 首页（主页面）

**路由**：`/`

**页面元素**：

- **顶部导航栏**：标题“AI 极简清单”，右侧显示网络状态图标（在线 / 离线）和设置入口。
- **输入区域**：单行或多行输入框，占位文本“写点什么... AI 自动分类”；右侧“添加”按钮（绿色、圆角、高可点击区域）。
- **分类标签栏**：横向展示标签：全部、购物、日程、其他、已完成。当前选中标签高亮显示。
- **清单列表容器**：
    - 独立滚动区域，仅列表区域可下拉刷新。
    - 每个事项展示为卡片，卡片内容：
        - 左侧：完成状态勾选框（未完成时空心圆，已完成时打勾绿色）。
        - 中间：事项内容（已完成时加删除线）、时间（若有，以小字体显示）。
        - 右侧：分类标签（购物/日程/其他），可选动态标签（如“工作”）。
    - **空状态**：当前分类无数据时，显示提示插图 + 文字“暂无事项，添加一条吧”。
    - **加载状态**：首次加载或切换分类时显示 loading 指示器。
- **首次启动提示**：首次打开网页时，从底部弹出轻提示“数据存于云端，换设备不丢失”，3 秒后自动消失，使用 `localStorage` 记录是否已展示。

**交互说明**：

- **添加事项**：
    - 用户输入文本，点击“添加”或按回车。
    - 调用前端 `parseInput` 解析分类和时间。
    - 前端通过 REST API 调用后端 `POST /api/todos` 写入数据库。
    - 添加成功后刷新当前列表；若当前分类不匹配，只更新数据源。
    - 清空输入框并保留继续输入的能力。
    - 输入为空时，提示“写点内容吧”。
    - 网络失败时，提示“网络开小差了，请稍后重试”。
- **标记完成 / 未完成**：
    - 点击卡片左侧勾选框，调用 `PATCH /api/todos/:id` 切换 `isCompleted`。
    - 如果当前分类不是“已完成”，事项变为已完成后从当前列表移除；在“已完成”标签中可查看。
- **长按菜单**：
    - 移动端 H5 通过 `touchstart` + `touchend` 实现长按，阈值为 `500ms`。
    - 达到阈值后弹出操作菜单：修改时间、修改分类、删除。
    - 若用户在 500ms 内松手或发生滑动，则取消长按触发。
- **删除事项**：
    - 长按卡片后点击“删除”，直接调用 `DELETE /api/todos/:id` 删除事项。
    - 删除成功后卡片从列表移除，并展示底部 Toast：“已删除”，附带“撤销”按钮。
    - 3 秒内点击撤销时，前端使用缓存的已删数据重新发起 `POST /api/todos` 恢复事项。
    - **无二次确认弹窗**，依靠撤销机制容错。
- **修改分类**：
    - 长按 -> 修改分类 -> 打开底部分类选择器（购物、日程、其他）。
    - 确认后调用 `PATCH /api/todos/:id` 更新 `category` 字段。
    - 若当前列表不再匹配该分类，则卡片从当前列表中移除。
- **修改时间**：
    - 长按 -> 修改时间 -> 打开时间弹窗。
    - 使用 `<input type="time">` 或自定义滚轮时间选择器。
    - 确认后调用 `PATCH /api/todos/:id` 更新 `time`、`displayTime`、`dateLabel`。
- **切换分类标签**：
    - 点击标签后，列表重新按条件加载对应数据。
    - 排序规则由用户偏好（上班族 / 宝妈）决定。
- **列表区下拉刷新**：
    - 仅在清单列表容器上实现下拉刷新，不带动整个页面下拉。
    - 用户下拉达到阈值后释放，触发重新获取最新列表数据。

### 4.2 设置页

**路由**：`/settings`

**页面元素**：

- 用户类型选择：上班族 / 宝妈（单选）。
- 动态标签开关：是否展示“工作”“育儿”等动态标签。
- 使用说明弹窗：展示典型输入示例和功能提示。

**交互说明**：

- 用户从首页右上角进入设置页。
- 修改设置项后立即生效。
- 偏好通过 `localStorage` 存储：`userType`、`dynamicTagsEnabled`。

### 4.3 修改时间弹窗

- 触发方式：长按菜单点击“修改时间”。
- 组件形式：底部弹层或中部 Modal。
- 输入方式：优先使用原生 `<input type="time">`，可替换为自定义滚轮时间选择器。
- 操作按钮：取消、确认。

## 5. 数据字段定义

### 5.1 前端本地数据结构（事项对象 Todo）

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | String | 是 | 后端返回的事项唯一标识 |
| `content` | String | 是 | 事项内容，最长 200 字 |
| `category` | String | 是 | 枚举值：`shopping` / `schedule` / `other` |
| `time` | String | null | 否 | 解析得到的时间，格式如 `15:00` |
| `displayTime` | String | null | 否 | 面向界面的展示时间，如“今天 15:00” |
| `dateLabel` | String | 否 | 日期标签，如“今天”“明天”“周三” |
| `isCompleted` | Boolean | 是 | 是否已完成 |
| `createdAt` | String | 是 | 创建时间 ISO 字符串 |
| `updatedAt` | String | 是 | 更新时间 ISO 字符串 |
| `dynamicTag` | String | null | 否 | 前端计算得到的动态标签 |

### 5.2 后端数据结构（MongoDB `todos` 集合）

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 自动 | MongoDB 自动生成 |
| `userId` | String | 是 | 用户标识，由前端生成并通过 `X-User-Id` 传递 |
| `content` | String | 是 | 事项文字内容 |
| `category` | String | 是 | `shopping` / `schedule` / `other` |
| `time` | String | null | 否 | 结构化时间值 |
| `displayTime` | String | null | 否 | 面向前端展示的时间文案 |
| `dateLabel` | String | 否 | 日期标签 |
| `isCompleted` | Boolean | 是 | 默认 `false` |
| `createdAt` | String | 是 | 服务器生成的 ISO 时间 |
| `updatedAt` | String | 是 | 最后更新时间 |

**索引建议**：`userId + createdAt`、`userId + category`、`userId + isCompleted`

### 5.3 本地存储（`localStorage`）

| 键名 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `userId` | String | 自动生成 | 当前设备用户标识（不同设备不同） |
| `userType` | String | `上班族` | 用户类型偏好 |
| `dynamicTagsEnabled` | Boolean | `true` | 是否展示动态标签 |
| `hasShownFirstTip` | Boolean | `false` | 是否展示过首次提示 |

## 6. 业务规则

### 6.1 AI 解析规则（前端函数 `parseInput`）

**输入**：用户输入的文本字符串。

**输出**：`{ content, category, time, displayTime, dateLabel }`

**规则优先级**：

1. **时间检测**：优先检测时间表达，如 `3点`、`14:30`、`下午三点`、`明天`、`后天`、`周一` 等。
2. **日程归类**：如命中时间相关表达，默认优先归类为 `schedule`。
3. **购物关键词检测**：若未命中明确时间，且文本包含 `买`、`购`、`囤`、`下单`、`超市`、`奶粉` 等关键词，则归类为 `shopping`。
4. **默认规则**：以上都未命中时，归类为 `other`。
5. **时间提取简化**：V1.0 保存结构化时间和展示时间，不要求在所有场景下转成绝对日期时间戳。

### 6.2 排序规则

根据 `userType` 决定未完成事项列表的排序：

- **上班族**：日程类优先，按 `time` 升序排列；购物类和其他类按 `createdAt` 降序。
- **宝妈**：购物类优先，按 `createdAt` 降序；日程类和其他类按 `createdAt` 降序。

已完成事项统一按 `updatedAt` 或 `createdAt` 降序展示。

### 6.3 动态标签规则

当 `dynamicTagsEnabled === true` 时：

- 如果 `category === 'shopping'` 且内容包含“奶粉”“尿布”“玩具”等关键词，则显示“育儿”标签。
- 如果 `category === 'schedule'` 且内容包含“会议”“汇报”“面试”等关键词，则显示“工作”标签。
- 标签仅作为辅助信息展示，不参与主排序。

### 6.4 用户标识规则

- 前端首次打开时生成一个 `userId`，存储在 `localStorage`。
- 所有接口请求头都带上：`X-User-Id: <localStorage中的userId>`。
- 后端仅返回和操作当前 `userId` 对应的数据。
- **不同设备生成的 `userId` 不同，数据天然隔离，不提供跨设备同步**（如需同步属于 V2.0 账号体系规划）。

## 7. 异常处理

| 场景 | 处理方式 |
| --- | --- |
| 网络断开，请求失败 | 提示“网络开小差了，请检查网络后重试” |
| 后端返回错误 | 提示“操作失败，请稍后再试” |
| 输入内容为空 | 提示“请写下你要做的事情” |
| 删除后撤销失败 | 提示“撤销失败，请手动重新添加” |
| 时间格式无法识别 | 保留原文内容，按默认分类处理 |
| 修改时间 / 分类失败 | 弹出轻提示“修改失败，请重试” |

## 8. 非功能需求

- **性能**：首屏可交互时间控制在 2 秒内；列表滚动流畅；常规接口响应时间 < 1 秒（正常网络）。
- **兼容性**：优先支持 iOS Safari、Android Chrome、微信内置浏览器等主流移动端浏览器；适配常见移动端分辨率。
- **隐私**：不依赖第三方社交登录；使用 `X-User-Id` 做前端用户隔离；敏感连接信息仅保存在服务端环境变量中。
- **可用性**：核心操作（添加、完成、修改、删除）在移动端 3 步内完成；长按、点击、下拉刷新等交互需有明确反馈。

## 9. 版本规划

- **V1.0（当前）**：实现全部 P0 + P1 功能（AI 解析、增删改查、分类筛选、排序、动态标签、设置页、撤销删除等）。
- **V2.0**：加入智能提醒（浏览器通知）、模糊检索、数据导出、主题模式等。
- **V2.1**：可选账号体系与跨设备同步（需要登录支持）。
